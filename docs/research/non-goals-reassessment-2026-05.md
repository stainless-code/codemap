## 0. Reframing question

## 2. Non-goals worth challenging â€” analytical history

These were defensive choices made when the project was small. Each subsection preserves the **original framing** and the **what's actually true** reasoning that motivated the flip; verdicts have lifted to canonical homes.

### 2.1 "No FTS5 / use ripgrep for full-text"

**Original framing:** "Full-text search across all file bodies â€” use ripgrep / IDE / opt-in FTS5".

**What's actually true:** SQLite has FTS5 built in. We already ship a `WITHOUT ROWID` table (`coverage`); a `source_fts` virtual table indexed at parse time is structurally identical. ripgrep's "loss" is **JOIN composition** â€” once content is an FTS table, you can write:

```sql
-- "TODO comments inside @deprecated functions in files with <50% coverage"
SELECT m.file_path, m.line_number, m.content
FROM markers m
JOIN symbols s ON s.file_path = m.file_path
              AND m.line_number BETWEEN s.line_start AND s.line_end
LEFT JOIN coverage c ON c.file_path = s.file_path AND c.name = s.name AND c.line_start = s.line_start
WHERE s.doc_comment LIKE '%@deprecated%'
  AND COALESCE(c.coverage_pct, 0) < 50
  AND m.kind = 'TODO';
```

ripgrep can't compose with `symbols` / `coverage` / `markers` in one shot â€” it can produce a list of file paths the agent then has to JOIN in JS. Same anti-pattern C.11 fixed for coverage.

**Lifted to:** [`roadmap.md Â§ Non-goals (v1) â†’ Floors`](../roadmap.md#floors-v1-product-shape) (FTS5 row, default-OFF policy preserved); shipped behavior documented in [`README.md Â§ CLI`](../../README.md#cli).

### 2.2 "No visualisation â€” rendering belongs to consumer"

**Original framing:** "Visualisation â€” skyline / ASCII art / animated diagrams; the index emits structured rows, rendering belongs to the consumer."

**What's actually true:** the non-goal conflates **rendering** (drawing pixels) with **shaping data into render-ready formats**. We already emit SARIF (a renderer-specific JSON format) â€” Mermaid / D2 / Graphviz are the same shape (edges + nodes), just different consumers. A `--format mermaid` formatter on `impact` / `dependencies` recipes lets agents include diagrams inline in chat / PR comments without a second tool.

**Lifted to:** [`roadmap.md Â§ Non-goals (v1) â†’ Floors`](../roadmap.md#floors-v1-product-shape) ("No renderer runtime" â€” formatters are shape-only).

### 2.3 "No static analysis"

**Original framing:** "Static analysis â€” dead code, duplication, complexity, architecture-boundary detection, fix actions are a different product class" (with `knip` / `jscpd` examples).

**What's actually true:** we already ship `deprecated-symbols`, `untested-and-dead` (post-C.11), `barrel-files`, `worst-covered-exports`, `visibility-tags` â€” those **are** static analysis. The original line was rhetorical (we don't ship a rule engine with severity levels and `// codemap-disable-next-line` comments) but the bullet over-asserted.

**Caveat (caught by triangulation):** `fan-in` and `fan-out` are **hotspot rankers**, not dead-code detectors â€” `fan-in.sql` literally `ORDER BY fan_in DESC LIMIT 15` ([source](../../templates/recipes/fan-in.sql)). They're structural-property recipes (legitimate static analysis), but they don't cover the **closed-dead-subgraph case**: an N-file pack where every file imports a sibling, so each has non-zero `dependencies` fan-in, but **none is reachable from any real entry point**. Closing that gap requires entry-point awareness â€” see [`plans/c9-plugin-layer.md`](../plans/c9-plugin-layer.md), not the "no static analysis" non-goal.

**Lifted to:** [`roadmap.md Â§ Non-goals (v1) â†’ Moats â†’ Moat A`](../roadmap.md#moats-load-bearing) ("verdicts are an OUTPUT mode, never a primitive") and the corresponding Floors row ("no opinionated rule engine / fix engine / severity levels").

### 2.4 "No persistent daemon"

**Original framing:** "Persistent daemon process â€” SQLite supports concurrent readers and our one-shot CLI startup is sub-100ms; revisit only if MCP / HTTP measurements demand it."

**What's actually true:** we **have** a daemon â€” `codemap watch`, `codemap mcp --watch`, `codemap serve --watch`. The non-goal preserved a constraint that no longer existed. The CLI cold-start argument still applies for one-shot `codemap query` invocations (preserved as a Floors row), but the long-running modes are explicitly daemon-shaped.

**Capability unlocked:** caching parsed ASTs in memory between requests would drop incremental reindex from ms to Âµs. Worth measuring; the data path already exists (we throw away the AST per request).

**Lifted to:** [`roadmap.md Â§ Non-goals (v1) â†’ Floors`](../roadmap.md#floors-v1-product-shape) ("No daemon by default" â€” opt-in for long-running modes); the **default `--watch` ON** flip is tracked in [`roadmap.md Â§ Backlog`](../roadmap.md#backlog) (resolved-but-unshipped per Q5 of the docs-capability-sync triangulation).

### 2.5 "No LSP replacement"

**Original framing:** "Replacing LSP or language servers â€” no rename / go-to-definition / hover types".

**What's actually true:** we have `show <name>` (file:line + signature), `impact <target>` (callers / callees), `watch` (live index) â€” capabilities **already in shipped engines** (`application/show-engine.ts`, `application/impact-engine.ts`, watch-mode chokidar) and **already exposed via MCP/HTTP/CLI**.

**Moat relation:** an LSP **engine** would re-extract structure inside the protocol layer â€” duplicating Moat B's substrate; that's the line. An LSP **diagnostic-push server** (different shape) is moat-A-aligned **iff** every diagnostic is a recipe rendering.

**Consumer-shape audit (2026-05, v3 â€” supersedes v1's "Very High agent UX" + v2's "shim deferred"):**

- **Agents** don't use LSP. They use MCP. The "Very High agent UX" framing in v1 was wrong.
- **VSCode / IDEs with `tsserver`** â€” `tsserver` dominates **standard pull-model requests** (`textDocument/definition` / `references` / `hover types` / `workspace/symbol`) for JS/TS users. A shim wrapping `show`/`impact`/`watch` engines for those requests would duplicate marginally and provide ~zero unique value. **This is what v2 correctly rejected.**
- **The actually-valuable LSP shape is a diagnostic-push server, not a request-handler shim.** Reference: fallow's `crates/lsp/` (~1800 LoC, `tower-lsp`) explicitly skips `definition` / `references` / `documentSymbol` and instead implements `diagnostic` (LSP 3.17 pull) + `code_action` + `code_lens` + `hover` + a custom `fallow/analysisComplete` notification consumed by their paired `editors/vscode/` extension.
- **Why this is moat-A-aligned (despite "diagnostics â‰ˆ verdicts"):** diagnostics here are an `--format lsp-diagnostic` rendering of bundled recipes. **Reviewer test:** "is the same finding queryable via `query --recipe X`?" If no recipe drives a diagnostic, it's a verdict-shaped primitive (rejected). Same discipline as `--format sarif` / `--format annotations` / `--format mermaid`.

**Lifted to:** [`docs/plans/lsp-diagnostic-push.md`](../plans/lsp-diagnostic-push.md) carries the canonical scope + publishing prerequisites + repo-structure tradeoffs.
